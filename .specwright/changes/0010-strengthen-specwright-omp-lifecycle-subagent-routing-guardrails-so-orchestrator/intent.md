## Approval notes
### Source request
Strengthen Specwright OMP lifecycle subagent routing guardrails so orchestrator prompts do not let weaker models skip required delegation and start doing worker tasks inline. Current context: when using models such as kimi-2.6, the receiving OMP agent may ignore the lifecycle spawn strategy and implement the task directly instead of spawning the configured specwright-* subagent. Existing Specwright routing is prompt-based because ExtensionApiLike only sends plain user messages and does not expose direct subagent dispatch. The current helper in src/core/prompts.ts says to use OMP's task tool but also includes a broad fallback: if the task tool, target agent, or model is unavailable, do the work directly in this agent. That broad fallback is too permissive for ADHD/non-compliant models. Goal: adapt the guardrail style to Specwright's simpler lifecycle without copying GSD prompts verbatim. Keep OMP-native prompt routing and generated .omp/agents model frontmatter. Make the top-level lifecycle prompt clearly define the receiving agent as an orchestrator for routed phases, not the worker. For research, plan, execute, and verify prompts, the first operational action should be spawning the matching configured Specwright subagent with the task tool when the task tool is available. Inline direct work must not be authorized just because the model wants to continue. The prompt must explicitly prohibit reading implementation files, editing code, running tests, updating artifacts, or reporting completion for the delegated lifecycle step while the subagent is active. It must require waiting for the subagent result before artifact/status updates or completion claims. It must list the exact target subagent name and configured model value. It must say to pass the full current lifecycle prompt/assignment to the subagent, including read-first files, rules, acceptance/checkpoint instructions, and any task-specific file list. It must also say not to ask the user to manually switch agents and not to spawn recursive lifecycle agents. Fallback behavior should be tightened: if task tool or target agent is genuinely unavailable, the orchestrator should report routing failure with the missing capability and only perform direct fallback when the lifecycle prompt explicitly permits it. For execute, default should be fail-closed rather than direct fallback because implementation conflicts are worse than pausing. For research/plan/verify, choose the safest Specwright-consistent behavior and document it in decisions.md; direct fallback may remain only if intentionally justified and made explicit, not broad/implicit. Update tests so printed research/plan/execute/verify prompts assert the stronger orchestration language, exact agent names, model values, wait-before-work rule, and no broad do-the-work-directly fallback. Preserve discuss, tasks, publish, handoff, and unrelated commands as direct workflows. Preserve generated agent frontmatter with model and spawns: [] from the prior smart-subagent-routing change. Acceptance: a future model reading a Specwright execute prompt should be told that inline implementation is not authorized while task tool routing is available; if routing cannot happen, it should report the routing failure instead of silently absorbing the executor role. Verification should include focused prompt tests and any relevant config/OMP agent tests if touched.
