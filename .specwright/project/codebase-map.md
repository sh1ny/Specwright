# Codebase Map

## Entry points

- `package.json` exposes the `specwright` bin and Bun scripts (`package.json:6-13`).
- `bin/specwright.mjs` is the published Node wrapper; it invokes `bun src/cli.ts` and fails clearly when Bun is absent (`bin/specwright.mjs:10-17`).
- `src/cli.ts` is the Bun CLI entrypoint; it prints help, delegates to `runSpecwrightCommand`, optionally prints generated prompts, and exits with the command result (`src/cli.ts:1-26`).
- `src/core/commands.ts` exports the core dispatcher `runSpecwrightCommand` and `renderHelp` (`src/core/commands.ts:1482-1517`).
- `src/runtime/omp/extension.ts` is the OMP extension entrypoint; generated `.omp/extensions/specwright/index.ts` re-exports it (`src/runtime/omp/extension.ts:7`, `src/runtime/omp/install.ts:31-38`).

## Core modules

- `src/core/commands.ts` — command kernel, argument parsing, scan/index refresh, command dispatch, and prompt selection. Public commands: `init`, `status`, `scan`, `new`, `discuss`, `research`, `plan`, `tasks`, `execute`, `verify`, `checkpoint`/`commit`, `handoff`, `pack`, `config`, `publish`, `complete` (`src/core/commands.ts:1482-1508`).
- `src/core/codebase-index.ts` — command-owned deterministic codebase map builder; discovers files through Git or filesystem fallback, classifies entrypoints/modules/tests, derives package scripts, records risks, and fingerprints indexed files.
- `src/core/state.ts` — default config/state, change validation, task parsing/sync, drift detection, config/state load/save, current-change resolution (`src/core/state.ts:57-87,124-279,299-421`).
- `src/core/validators.ts` — config validation, codebase-index validation, change validation, observed-output detection, validation report rendering (`src/core/validators.ts:73-99,100-308,408-493`).
- `src/core/prompts.ts` — runtime-neutral lifecycle, scan, discuss, retry, context-budget, checkpoint, and handoff prompt fragments.
- `src/core/git.ts` — shell-safe Git/GitHub helpers for branch creation, staging, commits, push, PR creation, base-branch resolution, and no-fast-forward merge.
- `src/core/json.ts` — JSON read/write helpers and file fingerprint computation.
- `src/core/paths.ts` — pure path helpers for `.specwright` and `.omp` layouts (`src/core/paths.ts:4-42`).
- `src/core/slug.ts` — slug and next-change-id helpers.
- `src/core/types.ts` — lifecycle, config, state, task, command, pack, workflow, and agent card types (`src/core/types.ts:1-132`).

## Runtime adapters

- `src/runtime/omp/extension.ts` registers the `/specwright` command, `specwright_status`, `specwright_checkpoint`, `specwright_validate`, the lifecycle delegation guard, and passive event hooks (`src/runtime/omp/extension.ts:24-160`).
- `src/runtime/omp/install.ts` generates `.omp/extensions/specwright/*`, `.omp/agents/specwright-*.md`, and `.omp/rules/specwright-workflow.md`; it skips existing user-owned files unless forced or targeted regeneration is requested (`src/runtime/omp/install.ts:140-187`).
- `src/runtime/omp/status.ts` computes read-only UI status from state, tasks, validation, and artifact fingerprints; it suppresses routine statuses and notifies for blocked/drift/checkpoint-needed states (`src/runtime/omp/status.ts:61-232`).
- `src/runtime/omp/prompts.ts` overlays OMP-specific lifecycle spawn, scout retry, scan parallelization, and discuss/ask guidance.
- `src/runtime/omp/args.ts` splits `/specwright` arguments while preserving quoted JSON config values.
- `src/runtime/omp/types.ts` defines structural OMP API types used by the adapter.

## Data and state artifacts

- `.specwright/config.json` — persisted config seeded by `defaultConfig`: lite/core/auto, 6 context files, 1200 output words, OMP enabled, autoCommit true, publishMode none, remote origin (`src/core/state.ts:57-87`).
- `.specwright/state.json` — current change and change cache (`src/core/types.ts:104-109`).
- `.specwright/project/*.md` — project intelligence files, including this map.
- `.specwright/project/codebase-index.json` — command-generated by `scan`; machine-readable map index with fingerprints validated by `validateCodebaseIndex` (`src/core/validators.ts:100-308`).
- `.specwright/changes/<id>-<slug>/*.md` — change artifacts generated from pack templates.
- `packs/core/pack.json` — built-in pack manifest with 13 artifacts, one workflow, four agents, one validator set (`packs/core/pack.json:1-33`).
- `packs/core/workflows/feature.json` — feature workflow; lite mode uses 10 artifacts and 6/1200 budget, full mode uses 13 artifacts and 12/3000 budget (`packs/core/workflows/feature.json:13-48`).

## Command surface

- `init` — create `.specwright`, copy core pack, create project files, install OMP adapter.
- `status` — report project/current change/task progress.
- `scan` — ensure/update project intelligence prompts and validate/refresh codebase map/index.
- `new` — create a numbered change from a kind and request.
- `discuss` — produce bounded clarification prompt and update lifecycle state.
- `research` — produce research prompt; OMP mode routes to `specwright-researcher`.
- `plan` — produce planning prompt; OMP mode routes to `specwright-planner`.
- `tasks` — produce CLI-parseable task prompt and sync task state.
- `execute` — select a task and produce task-scoped handoff; OMP mode routes to `specwright-executor`.
- `verify` — sync task state, run validators, write `verify.md`; OMP mode routes to `specwright-verifier`.
- `checkpoint`/`commit` — stage explicit files and commit a phase/task checkpoint.
- `handoff` — write compact handoff and validate before completion.
- `pack` — copy pack artifacts by manifest name.
- `config` — get/set typed config keys (`src/core/commands.ts:263-371`).
- `publish` — final remote publish modes: `none`, `push`, `pr`.
- `complete` — final guard command modes: `none`, `push`, `pr`, `merge`.

## Conventions

- Runtime-neutral core; OMP-specific behavior stays under `src/runtime/omp/*`.
- CLI commands return `CommandResult` and may include a generated prompt; CLI prints the prompt only with `--print-prompt` (`src/cli.ts:15-24`).
- Config keys are whitelisted and typed; array values are JSON strings through `config set` (`src/core/commands.ts:263-371`).
- Tasks use exact checklist lines `- [ ] T###: Title` or `- [x] T###: Title`; sync preserves in-progress/blocked state only when titles match (`src/core/state.ts:124-279`).
- Index paths must be safe relative paths; missing files are warnings, unsafe paths/fingerprint shapes are errors (`src/core/validators.ts:166-187,278-299`).
- OMP lifecycle phases `research`, `plan`, `execute`, and `verify` are expected to delegate through the `task` tool to matching lifecycle agents (`src/runtime/omp/extension.ts:10-15,135-149`).

## Known risks and gaps

- Pack validator manifest drift: `packs/core/validators/core.json` lists `SW001`-`SW008`, but implementation emits `SW009` (`packs/core/validators/core.json:3-12`, `src/core/validators.ts:457-463`).
- Published bin still requires Bun at runtime; no built `dist/` Node CLI exists (`bin/specwright.mjs:10-17`).
- Generated `.omp/` files can drift from source; adapter version marker catches extension package staleness, but user-owned rules/agents are intentionally preserved unless force or targeted regeneration is used (`src/runtime/omp/install.ts:140-187`).
- `codebase-index.json` is command-owned; agents may summarize its facts in Markdown but must not hand-edit JSON or fingerprints.

## Open questions

- Should generated `.omp/` files be fingerprinted in the machine index, or should only their source generators be indexed?
- Should scan refresh preserve historical deltas instead of replacing current summary sections?
